## The architectural truth this session uncovered

The chain CEO-082 → CEO-095 → CEO-098 (plus the parallel session's CEO-092 work in `c345f2e` which added `createPublicClient` and migrated 8 data functions) has eliminated the auth-side cookie reads across the entire store. Verified post-deploy:

- `/api/reader/me` returns the expected anonymous payload
- SiteHeader renders correctly without server-side auth
- BookCard sizes, font self-hosting, homepage Promise.all, dedupe, priorityFirst all live and effective

But every one of /, /browse, /collections, /book/[slug] still serves `cache-control: private, no-cache, no-store` and `x-vercel-cache: MISS` on the second hit.

The reason is `i18n/request.ts` lines 21–22:

```ts
const cookieStore = await cookies();
const cookieLocale = cookieStore.get(LOCALE_COOKIE)?.value;
```

`getRequestConfig` runs on every server render that uses next-intl's `getTranslations`. `cookies()` from `next/headers` is a Next.js dynamic function — calling it forces dynamic rendering on the entire route, regardless of `revalidate`, regardless of all our other cookie eliminations.

Per the comment in `i18n/request.ts` itself, the cookie read is essentially dead code in the current single-locale launch state. CEO-101 is bounded ~30 minutes for the simple fix (always return DEFAULT_LOCALE, hide LanguageSwitcher unless multi-locale is enabled). After it lands, the entire preparatory chain finally produces edge-cached HTML.
